# Docs Center Design

> 为“脱脂牛马”提供一个独立的产品说明型内容中心，统一承接 changelog、规则说明、使用帮助和 FAQ。

## 背景

随着赛季、经济系统、咖啡打卡和后续团队动态逐步落地，产品已经不再适合只靠零散文案解释功能。

目前缺少一个统一位置来回答这些问题：

- 最近更新了什么
- 赛季和经济规则是什么
- 新成员第一次使用要先看什么
- 常见误解和常见问题去哪里查

这些内容不适合塞进团队动态，因为它们不是“事件时间线”；也不适合继续塞进各个零散页面，因为会让说明分散、口径不一致。

## 产品定位

文档中心负责承接“产品说明型内容”。

它回答的问题是：

> 这是什么？怎么用？最近改了什么？

它不回答：

- 最近团队发生了什么
- 哪个成员刚刚达成了什么里程碑
- 哪一份周报值得回看

这些事件沉淀统一进入团队动态，不进入文档中心。

## 设计目标

1. 为产品说明内容提供统一入口，不占用主导航 Tab。
2. 用清晰的信息架构承接 changelog、规则、帮助、FAQ。
3. 第一版不依赖数据库和后台 CMS，先用本地内容稳定落地。
4. 保持页面轻量、可读、易维护。
5. 让新用户和老用户都能快速找到需要的信息。

## 非目标

- 不把文档中心做成论坛、评论区或运营社区。
- 不承接团队事件时间线、周报归档和未读播报。
- 不在第一版做全文搜索、内容权限、多语言 CMS。
- 不做在线编辑后台或管理员富文本发布器。
- 不取代页面内必要的就地提示文案。

## 入口与路由

### 入口

- 入口优先放在 Profile 下拉中。
- 不占用主导航 Tab。
- 入口文案建议直接使用 `文档中心`。

### 路由

- 提供独立页面 `/docs`。
- 页面属于 `(board)` 路由组，保持登录后可访问的一致体验。

## 信息架构

第一版建议采用单页面分区或 tabs 结构，承接四类核心内容：

- `更新日志`
- `赛季规则`
- `使用说明`
- `常见问题`

推荐顺序：

1. `更新日志`
2. `赛季规则`
3. `使用说明`
4. `常见问题`

原因：

- changelog 适合老用户先看
- 规则和使用说明适合新用户补课
- FAQ 适合作为兜底解释层

## 页面结构

推荐页面结构：

1. 页面头部
2. Tabs 或分区导航
3. 当前分区正文

### 页面头部

头部至少包含：

- 标题：`文档中心`
- 简短说明：承接规则、说明、更新记录

头部不需要做成营销页，也不需要产品介绍长文。

### Tabs / 分区导航

页面内用 tabs 或段落导航切换内容。

要求：

- 当前分区高亮清晰
- 手机宽度下仍可点击和横向浏览
- 不依赖复杂动画

## 内容分区设计

### 1. 更新日志

用于承接版本变化和体验更新。

每条 changelog 建议包含：

- `date`
- `title`
- `summary`
- `bullets`
- `tags` 可选

展示规则：

- 按时间倒序展示
- 新内容优先出现在顶部
- 不要求版本号一定存在，但日期必须清晰

适合承接的内容：

- 新功能上线
- 关键交互调整
- 规则文案澄清
- 重要 bug 修复

不适合承接：

- 团队昨天谁拿了周报高光
- 日常打卡或咖啡事件

### 2. 赛季规则

用于解释赛季、我的银子、牛马金库、赛季收入、冲刺条等核心产品规则。

建议采用“规则块”形式，每块回答一个明确概念：

- 什么是“我的银子”
- 什么是“牛马金库”
- 什么是“赛季收入”
- 赛季怎么开始和结束
- 冲刺条满格后会怎样

规则页目标是统一口径，不是复述数据库字段。

### 3. 使用说明

用于帮助成员快速理解常见入口和操作流程。

第一版建议覆盖：

- 如何完成健身打卡
- 如何理解 Profile 下拉里的资产信息
- 如何看赛季冲刺条
- 管理员如何管理赛季
- 后续团队动态上线后，可补充“如何查看团队动态”

这里回答“怎么走流程”，而不是记录团队发生过什么。

### 4. 常见问题

FAQ 用于兜底解释最容易混淆的问题。

建议优先覆盖：

- 为什么今天打卡了但冲刺条没再涨
- 为什么没有赛季也能获得银子
- 牛马金库是不是公共钱包
- 赛季收入和我的银子是什么关系
- 为什么某些页面入口在下拉里而不是主导航

FAQ 应该短、直接、可扫读，不写成长文教程。

## 与其他模块的边界

### 与团队动态

- 文档中心：说明、规则、帮助、FAQ、changelog
- 团队动态：团队事件、周报、里程碑、时间线

团队事件沉淀不放文档中心，统一进入团队动态。

### 与战报中心

- 战报中心：分析、数据复盘、结果生成
- 文档中心：产品说明与规则解释

战报中心里的图表解释不应反客为主地搬进文档中心。

### 与共享看板

- 共享看板：轻量公告与便签
- 文档中心：长期有效、可维护的说明内容

## 内容维护方式

第一版建议使用仓库内本地内容，不接数据库。

原因：

- changelog、规则、帮助、FAQ 频率可控
- 开发维护成本最低
- 不需要后台编辑权限和审核流

推荐结构：

```text
app/(board)/docs/page.tsx
components/docs-center/DocsCenter.tsx
components/docs-center/DocsTabs.tsx
components/docs-center/DocsSection.tsx
content/docs-center/changelog.ts
content/docs-center/rules.ts
content/docs-center/help.ts
content/docs-center/faq.ts
```

维护原则：

- changelog 通过新增条目追加
- 规则、帮助、FAQ 通过本地内容模块更新
- 内容变更跟随正常 PR 流程

## 数据策略

文档中心第一版使用静态本地内容或本地配置，不新增 Prisma 模型。

这样可以保证：

- `/docs` 页面无需 API 才能工作
- 内容不会和业务数据强耦合
- 规则说明与团队动态时间线天然分层

## 交互规则

第一版交互应保持简单：

- 点击 Profile 下拉中的 `文档中心` 进入 `/docs`
- 页面内切换 tabs 或锚点分区
- 不引入评论、点赞、收藏
- 不做编辑态与阅读态切换

如后续需要更强维护能力，再单独做后台化 spec。

## 文案风格

文档中心内容应以中文为主，风格要求：

- 直接
- 清楚
- 统一
- 不装腔

避免：

- 过长产品宣传文案
- 和页面 UI 提示重复堆砌
- 技术实现细节直接暴露给普通用户

## 测试建议

至少覆盖：

1. `/docs` 页面能正常渲染
2. 四个核心分区或 tabs 都能访问
3. changelog 按时间倒序展示
4. 规则、帮助、FAQ 内容可在手机宽度下正常阅读
5. 文档中心不依赖数据库或额外 API
6. 团队动态相关事件内容不会混进文档中心主结构

## 验收标准

- 用户能从 Profile 下拉进入 `/docs`。
- changelog、规则、帮助、FAQ 的边界清楚。
- 第一版内容可通过本地文件维护，不需要数据库。
- 文档中心和团队动态职责不混淆。
- 页面在桌面和移动端都保持可读。
