# Team Dynamics Design

> 为“脱脂牛马”补一条独立于战报中心和文档中心的团队事件时间线，让值得回看的团队变化有稳定入口、清晰沉淀和后续推送来源。

## 背景

当前产品已经有多块信息面：

- `健身打卡` 负责当天动作与结果
- `续命咖啡` 负责当天咖啡记录
- `共享看板` 负责轻量公告、便签和团队共用信息
- `战报中心` 负责复盘和汇总

但“最近团队发生了什么、哪些内容值得沉淀下来”仍然缺少独立承接面。现在的缺口主要有三类：

- 重要事件没有统一入口，用户需要分散到多个页面回忆
- 周报或赛季里程碑没有自然归档点，生成后难以回看
- 后续如果接企业微信群机器人推送，缺少一个稳定的上游内容源

这类问题不适合继续塞进 `战报中心`，因为战报中心偏分析和结果；也不适合塞进 `文档中心`，因为文档中心回答的是“这是什么、怎么用”，不是“最近发生了什么”。

## 产品定位

团队动态负责承接两类内容：

- 今天到最近几周内，团队发生的高价值事件
- 值得回看、值得沉淀、未来可能推送给外部沟通渠道的团队记录

它回答的问题是：

> 最近发生了什么？哪些事件值得团队回看？

它不回答：

- 产品规则怎么理解
- 功能怎么使用
- FAQ 或 changelog 去哪里看

这些说明型内容统一进入文档中心，不进入团队动态。

## 设计目标

1. 为团队动态提供明确入口，不新增主导航 Tab。
2. 建立独立于实时动作流的沉淀模型，避免时间线被噪音淹没。
3. 让周报、赛季里程碑、重要公告引用等事件可以统一回看。
4. 支持未读状态、时间线展示和基础筛选。
5. 为后续企业微信群机器人推送提供可复用事件源。

## 非目标

- 不把团队动态做成新的即时聊天或评论系统。
- 不承接 changelog、规则说明、帮助文档、FAQ。
- 不在第一版做复杂搜索、标签运营后台或内容审核流。
- 不在第一版接入企业微信 OAuth、成员绑定、指令回调。
- 不用团队动态替代共享看板、战报中心或文档中心。

## 入口与交互

### Navbar 入口

- 不新增主导航 Tab。
- 在 Navbar 中，优先放在头像左侧的喇叭入口。
- 喇叭显示未读数量徽标；无未读时只显示图标。

### 喇叭面板

点击喇叭后打开轻量时间线面板，职责是“快速看最近发生了什么”，而不是完整归档页。

第一版面板建议包含：

- 最近 `5-10` 条动态
- 未读强调样式
- 事件类型小标签
- 每条事件的时间、标题、摘要
- 一个“查看全部动态”入口，跳转 `/dynamics`

面板内不承载长文详情、复杂筛选和内容编辑。

### 全量页面

提供独立页面 `/dynamics`，用于完整时间线浏览。

页面承担：

- 全量时间线
- 事件类型筛选
- 未读 / 全部切换
- 周报、赛季里程碑等沉淀内容的回看

## 信息架构

推荐拆成两层：

1. 喇叭面板：最新动态预览
2. `/dynamics` 页面：完整时间线与筛选

这样可以同时满足：

- 高优先级入口足够近
- 归档内容有独立空间
- 不把 Navbar 下拉面板做成“半个复杂页面”

## 什么事件值得沉淀

团队动态不是原始事件流水，而是“经过筛选后值得回看”的事件流。

判断标准：

- 对团队整体有意义，而不是只对单个瞬时动作有意义
- 过几天再看仍然有价值
- 有可能被周报、月报或外部推送复用

### 第一版纳入的事件类型

- `WEEKLY_REPORT_CREATED`
  - 手动生成的周报
- `SEASON_STARTED`
  - 新赛季开启
- `SEASON_TARGET_REACHED`
  - 赛季冲刺条达标
- `SEASON_ENDED`
  - 赛季结束
- `TEAM_FULL_ATTENDANCE`
  - 团队全勤日
- `STREAK_MILESTONE`
  - 个人连续打卡达到明确里程碑，例如 `7 / 14 / 30`
- `COFFEE_SUMMARY`
  - 以摘要形式沉淀的咖啡事件，而不是每一杯都进时间线
- `BOARD_NOTICE_REFERENCE`
  - 共享看板中的重要公告或笔记引用

### 第一版不直接纳入的内容

- 每一次普通个人打卡都直接入时间线
- 每一杯咖啡都直接入时间线
- 系统帮助文档、规则说明、FAQ
- 纯界面改动的 changelog 文案

这些内容要么噪音太高，要么应由文档中心承接。

## 与现有模块的边界

### 与战报中心

- 战报中心：负责分析、生成、预览
- 团队动态：负责沉淀、展示、回看

手动生成的周报应该在战报中心发起，但结果保存到团队动态。

### 与共享看板

- 共享看板：轻量便签、公告、协作提醒
- 团队动态：可回看、可归档的事件时间线

共享看板里的重要公告可以被“引用进入团队动态”，但共享看板本身不迁移到团队动态。

### 与文档中心

- 文档中心：回答“这是什么、怎么用”
- 团队动态：回答“最近发生了什么”

产品说明型内容不放团队动态，统一进入文档中心。

## 数据模型

团队动态需要独立于实时 `ActivityEvent` 的沉淀模型。

原因：

- 实时动作流通常高频、细碎
- 沉淀时间线更关注价值密度和可回看性
- 后续推送不应该直接消费所有原始动作

### TeamDynamic

建议新增 `TeamDynamic` 作为核心事件表。

建议字段：

- `id`
- `teamId`
- `type`
- `title`
- `summary`
- `payloadJson`
- `actorUserId` 可空
- `sourceType` 可空
- `sourceId` 可空
- `importance`
- `occurredAt`
- `createdAt`
- `updatedAt`

字段语义：

- `type`
  - 事件类型，如 `WEEKLY_REPORT_CREATED`
- `title`
  - 时间线主标题
- `summary`
  - 列表摘要
- `payloadJson`
  - 结构化快照，用于周报、赛季事件等详情渲染
- `sourceType + sourceId`
  - 指向来源对象，如 `season / weekly-report / board-note`
- `importance`
  - 例如 `normal / high`
- `occurredAt`
  - 事件真正发生时间，不等同于记录写入时间

### TeamDynamicReadState

未读是“用户维度”状态，不应直接写在 `TeamDynamic` 表上。

建议新增 `TeamDynamicReadState`：

- `id`
- `teamDynamicId`
- `userId`
- `readAt`

约束：

- `@@unique([teamDynamicId, userId])`

这样可以支持：

- 每个成员独立的未读状态
- 喇叭徽标统计
- 未读 / 全部筛选

## 事件生成规则

### 自动生成

以下事件适合系统自动写入 `TeamDynamic`：

- 新赛季开启
- 赛季目标达成
- 赛季结束
- 团队全勤日
- 达到明确阈值的 streak 里程碑
- 手动生成周报后的结果落地

### 手动生成

以下事件允许由显式动作触发：

- 周报生成
- 共享看板重要公告引用进入时间线

### 事件去重

同一来源对象不应重复生成多条语义相同的动态。

示例：

- 同一份周报只能生成一条 `WEEKLY_REPORT_CREATED`
- 同一个赛季的“达标”事件只写一次

## 未读策略

第一版未读逻辑以“看过事件详情或显式标记”为准，不采用“打开面板自动全部已读”。

原因：

- 喇叭面板只是快速预览
- 自动清空未读会让用户错过真正重要内容

推荐规则：

- 打开喇叭面板：不自动清空未读
- 点击进入某条动态：将该条标记已读
- `/dynamics` 页面提供“全部标为已读”

## 周报落点

手动生成周报的产品归属是战报中心，但内容落点是团队动态。

推荐流程：

1. 用户在战报中心触发“生成本周战报”
2. 系统基于当前周数据生成结构化周报
3. 将周报快照写入 `TeamDynamic`
4. 团队动态时间线展示该周报卡片
5. 后续企业微信推送直接复用该条动态内容

这保证：

- 生成入口在分析场景里
- 沉淀入口在事件时间线里
- 推送来源稳定

## 推送衔接

团队动态应作为企业微信群机器人推送的上游内容源，而不是在推送时再临时拼文案。

第一版只需为后续推送保留接口，不需要真正接通推送。

设计要求：

- 时间线事件具备稳定 `type`
- 周报和高价值里程碑有结构化 `payloadJson`
- `importance` 足以支持后续推送筛选

后续推送系统可以基于 `TeamDynamic` 做：

- 手动推送某条动态
- 自动推送某类高价值事件
- 推送结果记录

## 页面与组件边界

推荐模块边界：

```text
app/(board)/dynamics/page.tsx
components/navbar/TeamDynamicsBell.tsx
components/team-dynamics/TeamDynamicsPanel.tsx
components/team-dynamics/TeamDynamicsTimeline.tsx
components/team-dynamics/TeamDynamicsFilters.tsx
components/team-dynamics/TeamDynamicCard.tsx
app/api/team-dynamics/route.ts
app/api/team-dynamics/read/route.ts
lib/team-dynamics/*
```

职责建议：

- `TeamDynamicsBell.tsx`
  - Navbar 入口
  - 未读数展示
  - 打开轻量面板
- `TeamDynamicsPanel.tsx`
  - 最近动态预览
  - “查看全部动态”
- `page.tsx`
  - 完整时间线页面
- `route.ts`
  - 列表读取，支持分页与筛选
- `read/route.ts`
  - 标记已读

## API 与读取策略

不建议把团队动态预加载进 `(board)/layout.tsx` 的初始大状态中。

原因：

- 团队动态是独立模块
- 未读与分页需求更像按需读取
- 周报和时间线内容可能持续增长

第一版建议：

- Navbar 喇叭单独请求最近动态与未读数
- `/dynamics` 页面按需请求时间线

## 测试建议

至少覆盖：

1. 喇叭显示未读徽标
2. 面板只展示最近若干条动态
3. `/dynamics` 页面可按事件类型筛选
4. 周报生成后被保存为团队动态
5. 相同来源对象不会重复生成重复动态
6. 未读状态按用户独立统计
7. 普通产品说明内容不会进入团队动态

## 验收标准

- 用户能从 Navbar 喇叭快速感知最近团队事件。
- 团队动态与文档中心、战报中心的职责边界清楚。
- 周报生成后可以在团队动态中回看。
- 未读数量和时间线筛选能工作。
- 数据模型足以支持后续企业微信群机器人推送。
