# Gamification Update Design

> 为“脱脂牛马”补一条以日常动力为核心的游戏化激励主线，用轻量任务、抽奖券、道具和暴击增益，把“每天愿意打开并完成健身打卡”做成更有期待感的日常行为。

## 背景

当前产品已经有健身打卡、赛季进度、个人资产、咖啡记录和战报能力，但“为什么用户每天都想回来完成打卡”这件事，仍然主要依赖自觉。

现有赛季系统已经提供了基础反馈：

- 完成健身打卡可以推进赛季冲刺条
- 完成打卡可以获得个人资产
- 团队可以在看板和战报里看到阶段性结果

但这些反馈还偏“结果导向”，缺少一层更日常、更轻、更有期待感的激励机制。尤其在小团队、熟人场景下，用户并不需要复杂监管，更需要一种：

- 今天打开就有事可做
- 今天做完就有反馈可领
- 今天领奖后会期待明天再来

## 产品定位

本次“游戏化更新”不是单独加一个抽奖按钮，也不是把健身打卡做成独立小游戏。

它的定位是一条新的日常激励主线：

`每日轻任务 -> 达成发券条件 -> 领取抽奖券 -> 抽到资源 / 增益道具 -> 反哺健身打卡、赛季贡献和个人资产`

这条主线服务的核心目标只有一个：

> 提高每天打卡的动力，让用户更愿意每天回来打开产品并完成健身打卡。

## 设计目标

1. 让健身打卡继续作为第一主目标，而不是被新的玩法替代。
2. 提供一条足够轻的日常任务链，让用户每天打开后除了打卡还有第二件能顺手完成的事。
3. 用抽奖券和道具制造期待感，但不让经济系统失控。
4. 保持小团队信任场景，不引入重验证、重审计和复杂反作弊。
5. 为后续背包、道具、暴击、运营活动和周报内容提供统一上游。

## 非目标

- 不把系统做成强监管的习惯打卡平台。
- 不要求照片、定位、计时、审批或成员互审。
- 不在第一版做复杂商城、交易市场或等级体系。
- 不让四维任务替代健身打卡本身的核心地位。
- 不在第一版引入大量自动运营、复杂概率活动或无限叠加的倍率系统。

## 当前已确认的产品原则

### 1. 健身仍然是主线

健身打卡必须始终保持最高优先级。新系统的价值在于“促成健身打卡”，不是把注意力从健身转移到纯抽奖玩法。

因此奖励结构上应明确区分：

- `健身券`：由健身打卡产出
- `生活券`：由四维日常任务完整达成产出

### 2. 采用信任型自报，而不是强校验

当前团队规模较小，成员彼此认识，也有线下沟通场景。第一版四维任务采用“自报完成”更合理。

这意味着：

- 成员点击完成即可算完成
- 默认相信成员自觉
- 不做照片、地理位置、长时间计时或证明材料

为了避免过度空心化，可以保留很轻的痕迹感，例如：

- 简短状态词
- 一句备注
- 团队动态里展示当天完成情况

但这些都应是弱约束，而不是审核机制。

### 3. 游戏化风格明确走“搞笑摸鱼风”

这一版不走严肃健康打卡路线，而是用带一点办公室黑话和团队自嘲感的包装方式，降低执行负担，提升记忆点。

当前确认的四个维度语言如下：

- `把电充绿`
- `把尿喝白`
- `把事办黄`
- `把股看红`

这些维度本质上分别映射：

- 活动身体 / 起身走动
- 补水
- 闲聊放松 / 情绪释放
- 信息输入 / 学一点新东西

## 核心玩法结构

### 每日任务结构

每日进入后，用户会看到四个固定维度，但每个维度当天对应的具体任务卡不同。

推荐结构：

- 维度固定为 `4` 个
- 每个维度每日抽取 `1` 张任务卡
- 用户完成后手动点击完成
- 四个维度全部完成，获得 `1` 张生活券

这种结构比“每天固定四个完全一样的任务”更好，因为它能同时兼顾：

- 认知稳定：用户始终知道是同一套四维框架
- 内容变化：每天有新鲜感
- 维护可控：只需要扩任务卡库，不需要频繁改玩法

### 抽奖券结构

当前已确认的发券逻辑如下：

- 完成当天健身打卡，获得 `1` 张券
- 完成四个维度的全部任务，再获得 `1` 张券
- 每日上限暂定为 `2` 张券
- 抽奖券可以长期累积，不会当天过期
- 抽奖券进入背包，并支持单抽与十连抽
- 健身打卡获得的券如果已经被消费，则当天健身打卡不能撤销，只给出明确提示
- 已消费的健身券不补偿、不产生负债、不回滚抽奖结果

这个结构的好处：

- 健身始终占据最直接的奖励入口
- 四维任务作为补充激励，不会喧宾夺主
- 发券速度可控，不容易把经济系统直接打爆
- 支持用户攒券十连，形成中期期待

### 抽奖与奖励

抽奖券用于参与抽奖，目标不是保证收益，而是提供“今天有机会拿到更好东西”的期待感。

当前讨论中适合进入奖池的奖励类型包括：

- 个人资产，例如银子
- 双倍券 / 暴击券
- 未来可扩展的其他道具或商品

第一版建议优先做“少量高识别度道具”，不要把奖池做得过杂。

详细卡池草案独立记录在 `docs/superpowers/specs/2026-04-25-gamification-card-pools.md`，避免主设计文档被具体奖品列表淹没。

### 个人资产闭环

本次游戏化更新可以为“个人资产”补上明确用途。

个人资产不再只是展示数字，而应成为可消耗的游戏化资源：

- 可以购买额外抽奖机会
- 可以购买少量基础道具
- 可以参与后续活动或特殊兑换

推荐闭环：

`健身打卡 / 四维全成 -> 获得免费抽奖券 -> 抽奖 -> 获得银子或道具 -> 银子购买额外抽奖机会或基础道具 -> 道具反哺健身打卡和赛季贡献`

但这里必须避免“抽奖产出银子，银子无限买券，再继续抽奖”的自循环膨胀。

第一版建议设定：

- 免费券每日最多 `2` 张，来自健身打卡和四维全成
- 银子可以购买额外抽奖券
- 抽奖券价格暂定为 `40` 银子 / 张
- 付费券每日最多购买 `3` 张
- 抽奖可以产出银子，但长期期望值必须低于购券成本
- 十连抽优先消耗已有券，不足部分再用银子补齐
- 十连抽必须至少已有 `7` 张券，最多用当日购券额度补齐 `3` 张
- 十连抽至少保底 `1` 个实用道具或以上
- 十连抽保底不保证稀有暴击
- 稀有道具主要来自抽奖，不全部开放银子直购
- 银子商城第一版只卖少量基础道具，不卖最强暴击券

这样个人资产能进入闭环，但不会轻易形成无限套利。

### 暴击与增益方向

赛季贡献与个人资产都可以被后续的“暴击型道具”影响，但不建议直接做成完全随机的纯暴击系统。

更稳的方向是：

- 一部分来自抽奖得到的可控道具
- 一部分来自轻量随机事件
- 触发后同时影响赛季贡献和个人资产

例如：

- 双倍赛季贡献券
- 双倍资产收益券
- 同时作用于贡献和资产的复合增益券
- 瑞幸咖啡券等真实福利兑换

### 风险与限制

这类道具必须从一开始就考虑经济边界，否则会造成赛季和资产滚雪球。

后续设计时需要重点限制：

- 每日或每周使用次数
- 是否允许叠加
- 倍率上限
- 是否仅对当日真实健身打卡生效
- 是否在特定赛季阶段才可使用
- 真实福利兑换需要管理员线下确认，不应自动写入咖啡打卡记录

### 保护类道具方向

保护类道具不应伪造健身结果。它们只能保护连续性或体验，不能直接发放健身收益。

`健身请假券` 的当前建议规则：

- 当天使用后，不算作完成健身
- 不发放健身抽奖券
- 不发放个人资产
- 不推进赛季贡献
- 不触发双倍券或暴击券
- 保护连续记录不断联
- 冻结下一次真实健身打卡的奖励档位

示例：

如果用户本来今天真实健身可获得 `40` 银子，但当天使用了健身请假券，那么今天不获得 `40` 银子，也不推进赛季；明天真实健身时，仍按 `40` 银子的档位结算。

## 四维任务系统

### 维度设计

推荐把四个维度作为稳定的大类，不频繁修改名称与定位：

### 1. 把电充绿

定位：活动身体、起身走动、短时恢复。

适合的任务方向：

- 起身走动几分钟
- 拉伸肩颈
- 从久坐状态里短暂离开

### 2. 把尿喝白

定位：补水、照顾身体状态。

适合的任务方向：

- 喝一大杯水
- 上午主动补一次水
- 下午主动补一次水

### 3. 把事办黄

定位：闲聊、放松、吐槽、办公室情绪释放。

适合的任务方向：

- 和同事闲聊两分钟
- 说一句今天最烦的事
- 分享一个无关工作的轻松话题

### 4. 把股看红

定位：信息输入、学习一点东西、看一点新消息。

适合的任务方向：

- 看一篇短文章
- 收藏一个有用链接
- 分享一个今天学到的新消息

### 任务卡片库

为了支持轮换玩法，需要建立任务卡片库。

推荐采用：

- `4` 个固定维度
- 每个维度维护独立卡池
- 每天从每个卡池各抽 `1` 张

第一版不要求题库特别大，先用每个维度 `5-8` 张任务卡启动即可。

### 卡片结构建议

每张任务卡至少包含：

- `title`
- `dimension`
- `description`
- `effort`
- `scene`
- `repeatCooldown`
- `isWeekendOnly`

字段语义：

- `effort`
  - 轻 / 中
- `scene`
  - 办公室 / 居家 / 通用
- `repeatCooldown`
  - 控制近期重复出现频率
- `isWeekendOnly`
  - 区分工作日和周末任务

详细字段规范独立记录在 `docs/superpowers/specs/2026-04-25-gamification-content-schema.md`。

当前已确认 `design/game-card-design.md` 可作为任务卡文案和维度表达的参考材料，但最终实现应以字段规范文档为准。

### 抽取规则建议

第一版建议保持简单：

- 每天每个维度抽 `1` 张
- 当天可手动换一次任务
- 最近 `3-5` 天尽量不重复
- 工作日和周末可使用不同卡池

## 页面与模块建议

这条主线至少会长出两个独立承接面。

### 入口命名

当前已确认：

- 游戏化主入口命名为 `牛马补给站`
- 第一版采用聚合页承接任务、抽奖、背包和弱社交邀请
- 暂不把背包拆成独立主导航入口

### 1. 牛马补给站

这是日常主入口，用于完成四维任务、查看今日进度和领取生活券。

页面职责：

- 展示今日四个维度任务卡
- 展示各维度完成状态
- 展示四维全成后的领奖状态
- 和健身打卡形成互相加成关系
- 展示抽奖券余额和今日发券状态
- 提供单抽、十连和银子补齐入口
- 展示背包中的主要道具
- 展示我发出的、我收到的、待响应的弱社交邀请

这个页面回答的是：

> 我今天除了健身，还能顺手完成什么？今天还能领什么？

推荐页面结构：

1. 今日任务区：四个维度任务卡，完成后领取生活券
2. 今日奖励区：展示健身券、生活券是否已领取
3. 抽奖区：单抽、十连、券余额、银子补齐
4. 背包区：按类型展示道具
5. 社交邀请区：我发出的、我收到的、待响应的邀请

### 2. 背包系统

背包系统负责承接抽奖券和道具，不建议只做成一个小角标数字。

第一版可先作为 `牛马补给站` 内的 Tab 或区块存在，后续道具数量变多后再拆成独立 `/bag` 页面。

页面职责：

- 查看当前持有的抽奖券
- 查看当前拥有的暴击券 / 双倍券 / 其他道具
- 选择何时使用可控增益
- 查看部分奖励来源和使用说明

这个页面回答的是：

> 我现在手里有什么？什么时候该用？

### 背包基础规则

当前已确认：

- 抽奖券永久有效
- 所有道具永久有效
- 真实福利券也永久有效
- 背包通过使用次数、叠加限制和结算边界控制经济，而不是通过过期回收道具

### 道具使用与叠加规则

当前已确认：

- 所有影响收益的道具都必须由用户手动使用
- 暴击类道具不可叠加
- 小暴击和强暴击不能同时生效
- 健身请假券不能触发任何暴击
- 弱社交道具可以同时持有，但每日使用次数受限

### 今日生效规则

当前已确认：

- 收益类增益道具使用后绑定当天，以 Asia/Shanghai 日期为准
- 如果当天尚未完成真实健身打卡，道具进入“今日待生效”状态
- 用户当天完成真实健身打卡后，道具生效并结算额外收益
- 如果当天已经完成真实健身打卡，使用道具后可以立即对当天打卡补结算
- 道具只作用于当天真实健身打卡，不跨天自动延续到下一次健身
- 如果当天最终没有真实健身打卡，收益类增益不生效，也不触发请假券
- 收益类增益建议在真实结算成功时才扣减库存，避免用户提前使用后未健身导致误吞道具

### 使用次数限制

当前建议：

- 强暴击类道具：每人每周最多使用 `1` 次
- 小暴击类道具：每人每天最多使用 `1` 次
- 任务换班券：每人每天最多使用 `1` 次
- 弱社交点名类道具：每人每天最多发起 `2` 次
- 团队广播类道具：每队每天最多触发 `1` 次
- 瑞幸咖啡券：系统内不限制持有数量，兑换需要管理员线下确认

## 与现有系统的关系

### 与健身打卡

- 健身打卡仍然是主线
- 游戏化系统负责增强动力和回访频次
- 所有设计都应优先反哺健身，而不是分流注意力

### 与赛季系统

- 道具与暴击可以反哺赛季贡献
- 但倍率和结算边界需要严格控制
- 赛季页面后续应能解释增益效果的来源

### 与个人资产

- 抽奖和道具终于能给个人资产一个明确用途和价值锚点
- 资产不再只是数字，而会成为可积累、可期待、可消耗的资源基础

### 数据存储边界

- 内容定义放本地配置：四维定义、任务卡片库、抽奖奖池定义、道具定义、概率配置
- 用户状态进数据库：每日任务、完成状态、抽奖券、背包、抽奖记录、道具使用、社交邀请、真实福利兑换
- 抽奖券采用“余额 + 流水”双层模型，`ticketBalance` 用于展示，`LotteryTicketLedger` 用于审计和追溯

### 与团队动态 / 周报

这条线未来可以给团队动态和周报提供新内容源，例如：

- 谁抽到了稀有道具
- 哪天团队多人完成四维全成
- 某个成员连续多天完成游戏化任务

但第一版不要求把所有任务动作都写进时间线。

### 与企业微信

弱社交道具后续需要和企业微信群机器人打通。

当前已确认：

- 弱社交道具使用后，会通过企业微信群机器人发送提醒消息
- 道具本身永久有效，但发出的社交邀请当天有效
- 对方可以响应，也可以忽略
- 未响应邀请过了当天自动失效
- 企业微信只负责把提醒送到团队沟通场景，系统内仍需保留邀请状态和响应记录
- 第一版弱社交响应不发银子，只生成响应记录、展示素材和后续周报 / 动态素材
- 不把所有普通点名都沉淀进团队动态，只沉淀高价值或多人响应事件

## API 边界

API 按页面动作拆分，不按数据库表裸露 CRUD。

### 补给站总状态

```text
GET /api/gamification/state
```

返回 `牛马补给站` 首屏需要的状态：

- 今日四维任务
- 四维完成状态
- 今日健身券 / 生活券领取状态
- 抽奖券余额
- 银子余额
- 背包摘要
- 待生效道具
- 我收到的弱社交邀请
- 我发出的弱社交邀请

### 四维任务

```text
POST /api/gamification/tasks/complete
POST /api/gamification/tasks/reroll
POST /api/gamification/tasks/claim-ticket
```

`claim-ticket` 单独提供，四维全成后由用户显式领取生活券，不在完成第 4 个任务时自动发券。

### 抽奖

```text
POST /api/gamification/lottery/draw
```

`draw` 支持 `SINGLE / TEN`。十连补齐、每日购券上限、券和银子扣减都必须由服务端校验。

第一版不提供独立购券接口。银子购券只作为十连抽补齐逻辑存在，用户必须至少已有 `7` 张券，最多用银子补齐 `3` 张。

### 背包和道具

```text
POST /api/gamification/items/use
POST /api/gamification/items/cancel-pending
```

`items/use` 根据 `itemId` 自动判断是今日增益、弱社交、请假券还是真实福利，并进入对应逻辑。

### 弱社交邀请

```text
POST /api/gamification/social/respond
POST /api/gamification/social/cancel
```

企业微信发送由 `items/use` 触发后在服务端完成。

### 真实福利兑换

```text
POST /api/gamification/redemptions/request
POST /api/admin/gamification/redemptions/confirm
POST /api/admin/gamification/redemptions/cancel
```

用户申请兑换，管理员确认或取消。

第一版必须支持管理员取消兑换。只有 `REQUESTED` 状态可取消，取消后返还库存；`CONFIRMED` 状态不能取消。

### 管理和配置预览

```text
GET /api/admin/gamification/config
```

第一版可以只读，用于检查当前任务卡、奖池和道具定义是否加载正常。

### 第一版最小 API

第一版建议优先实现：

- `GET /api/gamification/state`
- `POST /api/gamification/tasks/complete`
- `POST /api/gamification/tasks/reroll`
- `POST /api/gamification/tasks/claim-ticket`
- `POST /api/gamification/lottery/draw`
- `POST /api/gamification/items/use`
- `POST /api/gamification/social/respond`
- `POST /api/gamification/redemptions/request`
- `POST /api/admin/gamification/redemptions/confirm`

## 分阶段建议

### Phase 1：最小可玩版本

先验证用户是否真的会每天回来做。

范围建议：

- 一个独立游戏化页面
- 四个固定维度
- 每个维度有基础任务卡池
- 自报完成
- 完成健身得 `1` 张券
- 四维全成得 `1` 张券
- 一个基础抽奖入口
- 奖池先只含银子和少量增益券

### Phase 2：背包与可控增益

在确认日常参与度成立后，再补资源层。

范围建议：

- 独立背包页
- 道具库存
- 双倍券 / 暴击券的使用流程
- 道具对健身打卡和赛季贡献的结算接入

### Phase 3：内容扩充与活动化

在玩法成立后，再扩内容和运营能力。

范围建议：

- 扩大任务卡片库
- 引入主题任务或节日活动
- 引入团队双倍日或管理员发起活动
- 把高价值结果接入团队动态和周报

## 当前待继续讨论的问题

以下问题尚未定稿，后续需要继续补充：

- 抽奖券是立即抽，还是可累积后批量抽
- 抽奖入口放在游戏化页、背包页，还是两边都可进入
- 奖池概率如何分层
- 暴击券是否只作用于下一次健身打卡
- 道具是否允许叠加
- 个人资产未来除了抽奖和道具，还要不要承接商城或兑换
- 四维任务是否需要备注、状态词或团队可见痕迹
- 首批任务卡片库的具体内容和文案风格

## 结论

“游戏化更新”可以作为后续一个独立的大版本来规划。

它不是孤立页面，而是一条新主线，核心是：

- 用四维任务提供轻量日常入口
- 用抽奖券提供即时反馈
- 用背包和道具提供可积累期待
- 用暴击和增益反哺健身打卡、赛季贡献和个人资产

后续 roadmap 应以这份设计为基础，继续拆分为：

- 游戏化打卡页
- 任务卡片库
- 抽奖系统
- 背包系统
- 增益与暴击结算
- 周报 / 动态联动
