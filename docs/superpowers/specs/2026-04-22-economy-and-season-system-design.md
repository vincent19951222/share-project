# Economy & Season System Design

> 为“脱脂牛马”定义一套清晰的双系统经济模型：个人资产负责长期激励，赛季冲刺负责团队协作。

## 背景

当前项目中的经济概念主要围绕 `coins / teamCoins / targetCoins` 展开，但这几个值在语义上混合了三类不同信息：

- 个人收益
- 团队阶段目标
- 打卡带来的即时奖励

这会让用户自然产生疑问：“我今天打卡得到的奖励，到底进了我自己的口袋，还是进了团队池子，还是两边都加？”

本设计的核心目标，是把“个人成长”和“团队推进”彻底拆开，避免概念冲突，同时保留持续打卡的成就感和团队协作感。

## 设计目标

1. 建立清晰的双系统：
   - 个人资产系统
   - 赛季冲刺系统
2. 让每日打卡同时产生个人激励和团队推进，但两套账彼此解耦。
3. 支持管理员手动开启和结束赛季。
4. 让赛季结果具备可回看、可排行、可展示的视觉表达。
5. 保持规则简单，优先服务当前 5 人团队场景，不提前为更复杂组织结构过度设计。

## 非目标

- 本设计不引入可消费商城、下注、兑换道具等钱包玩法。
- 本设计不支持管理员自定义颜色主题。
- 本设计不处理超过 5 名成员的赛季配色策略。
- 本设计不实现复杂账本或多来源经济流水；当前先围绕打卡和赛季统计落地。

## 核心决策

| 维度 | 决策 | 说明 |
|------|------|------|
| 经济模型 | 双系统 | 个人资产与团队冲刺条彻底解耦 |
| 个人长期账户 | 永久累计 | 前台文案使用“我的银子” |
| 健身打卡总资产展示 | 保留牛马金库 | 仅展示所有成员个人总资产之和，不参与赛季结算 |
| 赛季内统计 | 单独统计 | 前台文案使用“赛季收入” |
| 团队目标 | 赛季冲刺条 | 前台文案使用“牛马冲刺条” |
| 赛季开启 | 管理员手动开启 | 不按自然月自动开启 |
| 赛季结束 | 管理员手动结束 | 即使未冲满也可结束并保留历史 |
| 赛季目标档位 | 固定档位 | 仅允许 `50 / 80 / 100 / 120 / 150` |
| 打卡奖励 | 连续递增 | `10 -> 20 -> 30 -> 40 -> 50`，第 5 天后维持 `50` |
| streak 归属 | 用户全局状态 | 不因赛季切换而重置 |
| 团队推进 | 每次有效打卡固定 `+1` 格 | 不受个人 streak 奖励倍率影响 |
| 冲刺条冲满后 | 赛季继续 | 保持满格展示，银子和赛季收入继续增长 |
| 赛季颜色 | 代码内置 | 每月固定底色 + 5 个固定贡献色 |
| 格子排序 | 按本赛季贡献降序分组 | 同分时使用稳定顺序避免跳动 |

## 概念定义

### 1. 我的银子

“我的银子”是用户的永久累计资产。

- 不会因赛季结束而清零
- 不会因管理员开启新赛季而重置
- 每次有效打卡都会增加
- 当前只承载“打卡长期激励”的角色，不承载消费功能

### 2. 牛马金库

“牛马金库”保留在健身打卡页面中，作为团队总资产展示。

- 它等于当前团队所有成员“我的银子”总和
- 它不是公共钱包
- 它不是赛季目标池
- 它不单独接收打卡奖励
- 它只是个人总资产的聚合展示结果

### 3. 赛季收入

“赛季收入”是用户在某一个赛季中获得的资产统计值。

- 不是第二个钱包
- 不是可消费余额
- 只用于赛季排行、赛季结算和历史展示
- 新赛季开启后从 0 开始重新统计

### 4. 牛马冲刺条

“牛马冲刺条”是一个赛季维度的团队目标进度组件。

- 不是公共钱包
- 不是资产池
- 只表示“这一期团队目标推进到了哪里”
- 由多个小格子组成，目标格数由管理员开赛季时选择
- 每次有效打卡推进 1 格

### 5. streak

`streak` 表示用户当前连续打卡天数，是用户级全局状态。

- 漏打一天直接断签
- 下一次打卡从第 1 天重新开始
- 不因赛季结束或新赛季开启而重置

## 打卡奖励规则

### 连续奖励

有效打卡奖励按当前连续天数结算：

| streak 天数 | 当次奖励 |
|------|------|
| 1 | 10 |
| 2 | 20 |
| 3 | 30 |
| 4 | 40 |
| 5+ | 50 |

补充约束：

- 连续打卡奖励只影响用户个人资产相关统计
- 它会增加“我的银子”
- 它会在 active 赛季中增加“赛季收入”
- 它不会让“牛马冲刺条”一次增加多格
- 无论 streak 多高，每次有效打卡对团队目标的推进都固定是 `+1` 格

### 断签规则

- 如果用户今天已经打过卡，则本次打卡无效
- 如果用户上次有效打卡是昨天，则 `streak + 1`
- 如果用户上次有效打卡不是昨天，则 `streak = 1`

### 结算时区

所有“是否同一天”“昨天/今天”的判断统一使用 `Asia/Shanghai`。

## 赛季规则

### 生命周期

赛季由管理员手动控制，不按自然月自动开启。

- 当没有 active 赛季时，管理员可以开启新赛季
- 开启时必须填写赛季目标名称，并选择目标格数档位
- 赛季进行中，管理员可以随时手动结束
- 结束后保留完整历史记录

### 目标格数

管理员开赛季时仅可选择以下固定档位：

- `50`
- `80`
- `100`
- `120`
- `150`

默认档位为 `50`。

赛季创建后目标格数锁定，不允许中途修改，避免进度口径变化。

### 目标命名

每个赛季在开启时都必须设置一个目标名称，用于说明这期大家在冲什么。

示例：

- `本月全勤冲刺`
- `五月掉脂挑战`
- `端午前不摆烂`

该名称在赛季创建后锁定，不允许中途修改。

### 冲刺条推进

如果当前存在 active 赛季，则每次有效打卡同时产生：

- “我的银子”增加当次奖励
- “赛季收入”增加当次奖励
- “牛马冲刺条”推进 1 格

如果当前不存在 active 赛季，则每次有效打卡只产生：

- “我的银子”增加当次奖励
- `streak` 正常延续或重置

此时不会产生赛季收入，也不会推进冲刺条。

### 冲满规则

当赛季冲刺条达到目标格数时：

- 冲刺条保持满格展示
- 赛季仍继续处于 active 状态
- 用户仍可继续打卡
- “我的银子”继续增长
- “赛季收入”继续增长
- 冲刺条不再继续增加额外格数

这意味着“冲满”只是团队里程碑，而不是赛季结束条件。

## 视觉与展示规则

### 月份底色

系统按月份为赛季提供固定底色方案，用于：

- 牛马冲刺条卡片背景
- 未完成格子的底色
- 标题、边框、高光等装饰

此方案由代码内置，不开放给管理员编辑。

### 冲刺条形态

“牛马冲刺条”在视觉上仍然是单条固定长度的进度条，但内部由多个小方格等比分割而成。

- 整体长度固定，不因档位变化而变长或变短
- 整体高度固定，不换行
- 支持 `50 / 80 / 100 / 120 / 150` 五个档位
- 小方格宽度由 `targetSlots` 自动等比分配
- 档位越高，单个方格越窄
- 组件不使用大间距断开，而使用轻量分隔线或细缝来保留“多个小格拼成一条进度条”的视觉感

推荐实现约束：

- 使用单行固定宽度容器
- 通过等比分栏渲染所有格子
- 避免 120 和 150 档位因为间距过大导致可用填充区域明显缩水
- 优先保证它看起来仍然是一条进度条，而不是一排离散按钮

### 成员贡献色

当前先按 5 人团队设计。

- 每个月固定一组底色
- 每个月同时固定 5 个与底色具备足够对比度的贡献色
- 成员颜色不需要管理员设置
- 开赛季后按固定顺序分配给成员
- 一个赛季中成员颜色保持稳定，不因排行变化而重新分配

### 贡献格子展示

牛马冲刺条中的已完成格子按“成员分组”展示，而不是按真实打卡时间顺序展示。

- 分组排序：按本赛季贡献格数从高到低
- 同分时：优先按 `firstContributionAt` 升序；若双方都尚未贡献，则回退到固定成员顺序
- 空格子：使用本月固定底色
- 已完成格子：使用成员本赛季贡献色

这样做的目标是：

- 一眼看出谁贡献最多
- 避免颜色在格子中来回跳动导致界面杂乱
- 保留“团队共同拼满冲刺条”的整体感

## 数据模型

本设计推荐引入显式赛季模型，而不是把所有字段都塞进 `User`。

### User

用户保留长期状态：

- `totalAsset`：个人总资产，对应“我的银子”
- `currentStreak`：当前连续打卡天数
- `lastPunchDate`：上次有效打卡对应的上海日期
- `role`：角色字段，用于识别管理员

说明：

- `currentStreak` 和 `lastPunchDate` 属于用户全局状态
- 它们不从属于某个赛季

### Season

赛季对象用于描述当前或历史赛季：

- `id`
- `monthKey`：例如 `2026-04`
- `goalName`
- `status`：`active | ended`
- `targetSlots`
- `filledSlots`
- `startedAt`
- `endedAt`

说明：

- `filledSlots` 上限为 `targetSlots`
- 即使赛季继续进行，也不会超过目标格数

### SeasonMemberStat

赛季成员统计用于承载赛季排行榜和冲刺条分组展示：

- `seasonId`
- `userId`
- `seasonAsset`：赛季收入
- `slotContribution`：本赛季贡献了多少格
- `colorIndex`：1 到 5，对应本月固定贡献色
- `firstContributionAt`：用于同分时稳定排序
- `memberOrder`：赛季开始时写入的固定成员顺序，用于零贡献或完全同分时回退排序

说明：

- 这张表是赛季 UI 的主要数据来源
- 它不是钱包，只是赛季统计对象
- 排序规则固定为：`slotContribution DESC -> firstContributionAt ASC -> memberOrder ASC`

### PunchRecord

打卡记录继续保留，但需要补充赛季结算信息：

- `userId`
- `seasonId`：当次打卡若发生在 active 赛季内则写入，否则允许为空
- `dayKey`：上海时区的日期键
- `streakAfterPunch`
- `assetAwarded`
- `countedForSeasonSlot`
- `createdAt`

说明：

- `dayKey` 用于确保“一天只能有效打卡一次”
- `seasonId` 为空的记录，表示发生在无 active 赛季时
- `countedForSeasonSlot=false` 的记录，通常出现在“赛季已满格但仍继续打卡”的场景

## 打卡结算流程

一次有效打卡的推荐结算顺序如下：

1. 以 `Asia/Shanghai` 生成今天的 `dayKey`
2. 检查今天是否已有有效打卡记录
3. 根据 `lastPunchDate` 计算新的 `streak`
4. 按 `streak` 映射出本次奖励金额
5. 更新 `User.totalAsset`
6. 更新 `User.currentStreak`
7. 更新 `User.lastPunchDate`
8. 写入 `PunchRecord`
9. 如果存在 active 赛季：
   - 更新 `SeasonMemberStat.seasonAsset`
   - 若冲刺条未满，则：
     - `Season.filledSlots + 1`
     - `SeasonMemberStat.slotContribution + 1`
     - 首次产生贡献时写入 `firstContributionAt`
10. 返回更新后的前台快照

### 无 active 赛季时的行为

即使管理员尚未开启赛季，用户也应该可以正常打卡并获得个人奖励。

这是为了确保：

- 个人激励系统始终在线
- 管理员操作不会阻断基础打卡行为
- streak 不会因为赛季管理节奏而被动中断

## 前台展示设计

首页或核心看板页应优先展示以下信息：

### 1. 我的银子

展示用户永久累计的个人资产。

### 2. 我的 streak

展示：

- 当前连续打卡天数
- 下一次有效打卡预计可得奖励

示例文案：

- `已连续 4 天`
- `明天继续可得 50 银子`

### 3. 牛马金库

展示当前团队所有成员“我的银子”总和。

说明：

- 它只用于展示团队累计资产规模
- 它不承担赛季目标语义
- 它和牛马冲刺条同时存在，二者不能混用

### 4. 牛马冲刺条

展示：

- 赛季目标名称
- 当前赛季目标格数
- 当前已完成格数
- 彩色贡献分组
- 当前赛季是否已达成目标

### 5. 赛季收入排行榜

按 `SeasonMemberStat.seasonAsset` 展示本赛季排行。

说明：

- 排行依据使用“赛季收入”，而不是“我的银子”
- 这样新老用户都可以参与当期竞争

## 管理后台最小功能

管理员后台先做一个轻量赛季管理页，只提供必要操作。

### 当前赛季卡片

展示：

- 当前是否存在 active 赛季
- 赛季月份
- 赛季目标名称
- 当前目标格数
- 当前完成格数
- 是否已满格
- 开始时间

### 开启新赛季

仅在当前没有 active 赛季时可用。

管理员需要选择：

- 赛季目标名称
- 目标格数档位：`50 / 80 / 100 / 120 / 150`

系统自动完成：

- 创建赛季记录
- 读取当月固定底色与 5 个贡献色方案
- 为当前团队成员初始化 `SeasonMemberStat`，并写入固定的 `memberOrder`

### 结束当前赛季

管理员可以在任意时刻手动结束当前赛季。

结束后：

- 当前赛季不再接收新的赛季收入累计
- 当前赛季不再推进冲刺条
- 历史记录永久保留
- 用户个人银子和 streak 不受影响

### 赛季历史

历史页至少应支持查看：

- 每期赛季的目标名称
- 每期赛季的目标格数
- 实际完成格数
- 是否冲满
- 前几名赛季收入
- 每位成员的贡献格数

## 错误处理与边界情况

### 重复打卡

- 同一用户同一天重复打卡必须被拒绝
- 不得重复增加银子
- 不得重复推进冲刺条

### 赛季未开启

- 用户仍可有效打卡
- 只结算个人银子和 streak

### 赛季已满格

- 用户仍可有效打卡
- 继续增加“我的银子”和“赛季收入”
- 不再增加 `slotContribution`
- 不再增加 `filledSlots`

### 赛季提前结束

- 允许管理员在未满格时结束赛季
- 历史页面保留当时的真实进度和排行

### 赛季切换

- 新赛季开始时，`赛季收入` 和 `slotContribution` 从 0 开始
- `我的银子`、`currentStreak`、`lastPunchDate` 保持延续

## 测试建议

至少覆盖以下场景：

1. 用户首次打卡获得 10 银子
2. 连续 2 到 5 天打卡奖励递增
3. 第 6 天及以后持续获得 50 银子
4. 漏打一天后 streak 重置为 1
5. 同一天重复打卡不会重复结算
6. 无 active 赛季时，仍能正常增加我的银子
7. 有 active 赛季时，打卡同时增加赛季收入和冲刺条
8. 冲刺条满格后，赛季收入继续增加但冲刺条不再增长
9. 管理员可提前结束未满格赛季
10. 新赛季开始后，用户 streak 延续，但赛季收入重新从 0 统计
11. 冲刺条分组顺序按贡献降序展示，同分时顺序稳定
12. 当月固定底色和 5 个贡献色能正确映射到前台组件

## 迁移建议

推荐按以下顺序落地：

1. 引入赛季相关数据模型和用户长期状态字段
2. 重构打卡结算逻辑，使个人银子和 streak 成为服务端权威状态
3. 新增管理员赛季管理 API 与后台页面
4. 将前台 `teamCoins` 迁移为“牛马金库 = 全员我的银子总和”，并新增独立的牛马冲刺条组件
5. 增加赛季历史和排行榜展示

## 最终结果

该设计将当前混合的“积分 / 金库 / 目标”语义，拆解为三条明确主线：

- “我的银子”负责个人长期成长
- “牛马金库”负责展示团队累计资产总和
- “赛季收入”负责当期竞争和排行
- “牛马冲刺条”负责团队阶段性协作目标

这三者共同服务于同一个打卡动作，但不再共用同一套账，从而避免用户理解冲突，并为后续赛季化运营提供清晰基础。
