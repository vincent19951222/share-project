# Manual Weekly Report MVP Design

> 为“脱脂牛马”补一条主线手动周报能力：由管理员在战报中心生成一份本周主线数据周报，先形成管理员私有草稿，再选择发布到团队动态，供全员回看。第一版不接 LLM，不依赖牛马补给站数据。

## 背景

当前主线已经有三块基础能力：

- `战报中心`：承接汇总、趋势和复盘
- `团队动态`：承接高价值事件的沉淀与回看
- `文档中心`：承接规则、说明、FAQ 等说明型内容

但“本周团队整体表现”的内容链路还没有打通：

- 战报中心能展示当下趋势，但没有一份可固化、可发布、可回看的周摘要
- 团队动态已经支持 `WEEKLY_REPORT_CREATED` 类型，但还没有主线周报来源
- 后续企业微信推送需要一个稳定、结构化、可复用的周报快照源

这件事不应该直接等同于 GM-15。GM-15 的重点是“牛马补给站游戏化周报”，依赖任务、抽奖、背包、道具和弱社交数据。主线手动周报 MVP 应先独立落地，打通：

```text
战报中心生成 -> 管理员私有草稿 -> 发布到团队动态 -> 全员回看
```

后续 GM-15 再扩充同一套周报结构，不重做主链路。

## 产品定位

主线手动周报回答的问题是：

> 这一周团队整体的健身、咖啡和赛季节奏怎么样？哪些结果值得沉淀和回看？

它不是：

- 自动定时周报系统
- 游戏化周报
- 企业微信推送系统
- 长图 / PDF / 海报导出功能

## 设计目标

1. 在 `战报中心` 提供主线手动周报生成入口。
2. 只有管理员能生成和发布周报。
3. 生成后的周报先形成管理员私有草稿，不直接暴露给成员。
4. 管理员确认后，将周报发布成一条 `WEEKLY_REPORT_CREATED` 的团队动态。
5. 发布后所有成员都能在 `团队动态` 中回看同一份周报快照。
6. 第一版不使用 LLM，周报总结文案采用规则模板生成。
7. 为 GM-15 预留扩展位，使游戏化数据未来可以接入同一套周报结构。

## 非目标

- 不做自动定时周报。
- 不做日报、月报或季度报。
- 不做企业微信推送。
- 不做长图、PDF 或图片导出。
- 不让普通成员查看未发布草稿。
- 不引入牛马补给站任务、抽奖、道具、弱社交等游戏化指标。
- 不新增复杂周报后台或编辑器。
- 不因为生成或发布周报而改变任何业务数据、奖励、银子、赛季进度或咖啡记录。

## 角色与权限

### 管理员

- 可以在战报中心看到 `生成本周周报` 按钮。
- 可以查看自己生成的当前周草稿。
- 可以重新生成，覆盖自己当前周的最新草稿。
- 可以点击 `发布到团队动态`。

### 普通成员

- 看不到草稿生成或发布按钮。
- 看不到未发布草稿。
- 只有在管理员发布后，才能通过团队动态查看周报结果。

### 草稿可见性

第一版采用最严格版本：

- 草稿仅对“生成它的管理员本人”可见。
- 不做管理员之间共享草稿。
- 已发布内容不再依赖草稿权限，统一通过团队动态公开回看。

## 入口与产品流

### 入口位置

- 入口放在 `战报中心`
- `团队动态` 不承担生成行为，只承担发布后的回看

这符合当前主线职责划分：

- `战报中心`：分析、生成、预览
- `团队动态`：沉淀、展示、回看

### 用户流

#### 1. 生成草稿

管理员在战报中心点击 `生成本周周报`：

1. 服务端按当前自然周聚合主线数据
2. 生成一份结构化周报草稿
3. 草稿保存为管理员私有数据
4. 战报中心展示草稿预览

#### 2. 发布到团队动态

管理员确认无误后点击 `发布到团队动态`：

1. 读取当前管理员的本周草稿
2. 将草稿快照发布为一条 `WEEKLY_REPORT_CREATED`
3. 使用 `sourceType + sourceId` 去重，避免同一周重复发布成多条
4. 发布成功后，团队所有成员可在团队动态中回看

#### 3. 成员回看

- 成员通过喇叭预览面板或 `/dynamics` 页面看到已发布周报
- 第一版不单独开 `/weekly-reports/:id` 页面
- 已发布内容统一由团队动态承接

## 周期规则

第一版只支持“当前自然周”。

规则：

- 时区统一使用 `Asia/Shanghai`
- 周一是一周开始
- 周日是一周结束
- 当周未结束时，周报按“周一到当前时间”计算
- `weekStartDayKey` 使用上海日期下的 `YYYY-MM-DD` 周一

示例：

```text
2026-04-27 周一 -> weekStartDayKey = 2026-04-27
2026-05-03 周日 -> weekEndDayKey = 2026-05-03
```

第一版不做历史周切换器，也不做自定义日期范围。

## 周报内容范围

第一版只统计当前主线已稳定存在的数据。

### 必含指标

- 本周打卡次数
- 本周全勤日
- 本周运动高光成员
- 本周咖啡王
- 团队低谷日和高峰日
- 当前赛季冲刺进度
- 一段简短团队总结文案

### 指标说明

#### 本周打卡次数

- 统计本周窗口内所有有效打卡记录
- 作为团队整体活跃度基础指标

#### 本周全勤日

- 指团队在本周内达成全员有效打卡的天数
- 用于表达团队协同节奏

#### 本周运动高光成员

- 从本周有效打卡和连续打卡表现中选出少量高光成员
- 第一版不做复杂评分系统，采用稳定、可解释的规则

#### 本周咖啡王

- 基于现有咖啡统计快照得出本周咖啡最活跃成员或团队咖啡摘要
- 只要当前数据源稳定即可，不要求引入额外咖啡报表表

#### 团队低谷日和高峰日

- 复用战报中心已有日级趋势视角
- 提取本周日级表现中最高点和最低点

#### 当前赛季冲刺进度

- 读取当前赛季状态与冲刺条进度
- 主要用于表达团队阶段位置，不引入额外赛季结算逻辑

#### 团队总结文案

- 第一版不调用 LLM
- 使用规则模板生成
- 模板基于本周指标组合出 1 到 2 句团队总结
- 后续可替换为“模板生成 + LLM 润色”的可选增强层

## 数据模型

第一版不新增复杂报表归档表，但需要一张轻量草稿表。

### WeeklyReportDraft

建议新增 `WeeklyReportDraft`：

- `id`
- `teamId`
- `createdByUserId`
- `weekStartDayKey`
- `summary`
- `payloadJson`
- `createdAt`
- `updatedAt`

字段语义：

- `teamId`
  - 草稿所属团队
- `createdByUserId`
  - 草稿生成管理员
- `weekStartDayKey`
  - 当前周标识，用于限制“一人一周一份草稿”
- `summary`
  - 草稿的简短摘要，便于战报中心快速展示
- `payloadJson`
  - 周报结构化快照，供预览与发布复用

约束建议：

- `@@unique([teamId, createdByUserId, weekStartDayKey])`

这意味着：

- 每个管理员每周只维护自己的一份最新草稿
- 重复生成时，覆盖当前周自己的草稿

### TeamDynamic 复用

已发布周报不单独新建公开周报表，统一写入现有 `TeamDynamic`：

- `type = WEEKLY_REPORT_CREATED`
- `sourceType = "weekly-report"`
- `sourceId = "<teamId>:<weekStartDayKey>"`
- `payloadJson` 保存发布时周报快照

这样可以保证：

- 团队动态天然支持回看
- 发布后内容和草稿权限解耦
- 后续企业微信或 GM-15 可直接复用这份发布快照

## 快照结构

第一版周报快照建议采用可扩展结构：

```ts
interface WeeklyReportSnapshot {
  version: 1;
  weekStartDayKey: string;
  weekEndDayKey: string;
  generatedAt: string;
  generatedByUserId: string;
  summary: string;
  metrics: {
    totalPunches: number;
    fullAttendanceDays: number;
    peakDay?: { dayKey: string; value: number };
    lowDay?: { dayKey: string; value: number };
    seasonProgress?: {
      filledSlots: number;
      targetSlots: number;
      status: string;
    };
  };
  highlights: {
    topMembers: Array<{ userId: string; label: string; value: string }>;
    coffee?: { userId?: string; label: string; value: string };
  };
  sections: Array<{
    id: string;
    title: string;
    summary: string;
    bullets: string[];
  }>;
}
```

关键点：

- `summary` 单独抽出，便于列表展示
- `metrics / highlights / sections` 为后续 GM-15 扩展预留稳定接口
- GM-15 未来只需要往 `sections` 或 `metrics` 增加游戏化分区，不必重做主链路

## 战报中心 UI

在现有 `ReportCenter` 中新增一个 `本周周报` 模块。

### 管理员态

展示：

- 周报模块标题
- 当前草稿状态
- `生成本周周报`
- `发布到团队动态`
- 草稿预览内容

建议状态：

- `未生成`
- `已生成草稿，未发布`
- `已发布到团队动态`

### 成员态

不展示生成和发布按钮。

可展示：

- 若当前周已发布：显示最近一次已发布摘要，并引导到团队动态查看
- 若当前周未发布：显示“本周周报尚未发布”

### 交互原则

- 生成是显式动作，不自动生成
- 发布是独立动作，不在生成后自动公开
- 发布成功后给出明确确认，并支持跳转到团队动态

## 团队动态展示

发布后的周报作为一条 `WEEKLY_REPORT_CREATED` 动态展示。

第一版卡片建议包含：

- 周范围
- 一句摘要
- 2 到 3 个关键数字
- 周报类型标签

在喇叭预览面板中：

- 只展示标题和摘要

在 `/dynamics` 页面中：

- 展示更完整的快照卡片
- 不要求独立详情页

## API 边界

### `POST /api/reports/weekly/draft`

职责：

- 仅管理员可调用
- 按当前自然周生成或覆盖自己的草稿
- 返回最新草稿快照

失败场景：

- 非管理员调用
- 周报聚合失败

### `GET /api/reports/weekly/draft`

职责：

- 仅管理员可调用
- 读取自己当前周草稿

失败场景：

- 非管理员调用
- 当前周无草稿

### `POST /api/reports/weekly/publish`

职责：

- 仅管理员可调用
- 读取当前管理员当前周草稿
- 发布为 `TeamDynamic`
- 按 `sourceType + sourceId` 去重

失败场景：

- 非管理员调用
- 无草稿时发布
- 动态写入失败

成员端不提供草稿 API。

## 汇总与文案生成策略

### 汇总层

第一版汇总全部走确定性计算：

- 直接读取现有主线数据
- 不调用 LLM
- 不引入不稳定评分模型

### 文案层

第一版总结文案使用规则模板：

- 根据打卡次数、全勤日、高峰日、低谷日、赛季进度等指标拼装
- 文案轻松但稳定
- 允许少量模板分支，不允许自由生成漂移

### 后续 LLM 升级预留

后续如果要升级到 LLM，只允许作为“可选文案增强层”：

- 输入：结构化周报快照
- 输出：润色后的总结文案
- 必须保留模板回退路径
- 不允许让 LLM 成为指标计算来源

## 与 GM-15 的关系

主线手动周报 MVP 和 GM-15 的关系是：

- 当前实现主线周报链路
- 后续 GM-15 只扩“内容”，不改“流程”

GM-15 可以复用：

- `WeeklyReportDraft`
- 发布到 `TeamDynamic` 的机制
- `WEEKLY_REPORT_CREATED` 展示链路
- 周报快照的 `metrics / highlights / sections` 结构

GM-15 未来新增的只是：

- 游戏化指标
- 游戏化高光
- 游戏化周报文案分区

而不是重做：

- 草稿权限
- 发布机制
- 团队动态回看

## 错误处理

- 生成失败不应影响战报中心其他模块
- 发布失败不应修改任何主线业务数据
- 无草稿时发布，应返回明确业务错误
- 周报模块局部错误应在模块内展示，不让整个战报中心空白

## 验收标准

1. 管理员可以在战报中心生成当前周主线周报草稿。
2. 草稿只对生成它的管理员本人可见。
3. 普通成员看不到草稿和生成 / 发布按钮。
4. 管理员可以将草稿发布为一条 `WEEKLY_REPORT_CREATED` 团队动态。
5. 同一团队同一周不会重复发布多条相同周报动态。
6. 发布后所有成员都能在团队动态中查看这份周报结果。
7. 第一版周报不依赖 LLM，不依赖牛马补给站数据。
8. 生成和发布周报不改变任何打卡、咖啡、赛季、银子或其他业务状态。

## 非第一版延伸

- 自动定时周报
- 历史周选择器
- 企业微信一键推送
- 长图 / PDF / 海报导出
- 管理员之间共享草稿
- 成员预览未发布草稿
- 游戏化周报内容（GM-15）
- LLM 文案增强
